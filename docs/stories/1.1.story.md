## Story

Como operador da All In,
quero uma demo estatica, visualmente impecavel e navegavel do cardapio digital,
para apresentar restaurantes brasileiros um fluxo completo de cardapio, carrinho e pedido simulado sem comissao por pedido.

## Business Value

A demo V1 permite validar o pitch comercial da All In antes da construcao SaaS completa. Ela mostra o valor central do produto: cardapio digital profissional, pedido online simulado e experiencia mobile-first sem depender de Supabase, Mercado Pago real ou painel admin.

## Acceptance Criteria

### AC1 - Projeto Next.js V1 inicializado

Given que o repositorio ainda nao possui app implementado,
When a story for desenvolvida,
Then deve existir um projeto Next.js 14 com App Router, Tailwind CSS, TypeScript e scripts para `lint`, `typecheck`, `test` e `build`.

### AC2 - Dados hardcoded da demo

Given que a V1 nao usa banco de dados,
When o cardapio for renderizado,
Then restaurante, categorias e pratos devem vir de `data/demo-restaurante.ts`.

### AC3 - Cardapio publico navegavel em `/demo`

Given um usuario acessando `/demo`,
When a pagina carregar,
Then deve exibir header sticky, hero, barra de categorias sticky, secoes por categoria, cards normais e cards de destaque.

### AC4 - Interacao com pratos

Given um prato disponivel no cardapio,
When o usuario clicar no card,
Then deve abrir modal com foto, nome, descricao, ingredientes, seletor de quantidade e CTA de adicionar ao pedido.

### AC5 - Carrinho local funcional

Given itens adicionados ao pedido,
When o usuario abrir o carrinho,
Then deve ver itens, quantidades, subtotais, total e acoes para alterar ou remover itens.

### AC6 - Checkout simulado V1

Given que `NEXT_PUBLIC_PAGAMENTO_REAL=false`,
When o usuario preencher nome e telefone e confirmar o pedido,
Then a aplicacao deve simular processamento, gerar numero de pedido e exibir tela de confirmacao sem redirect externo e sem cobranca real.

### AC7 - WhatsApp interno da All In

Given um pedido simulado confirmado,
When a API `/api/pedido-simulado` for chamada,
Then deve tentar enviar mensagem para o grupo WhatsApp interno da All In usando Evolution API, com prefixo `[DEMO]` e aviso explicito de pagamento simulado.

### AC8 - Falha no WhatsApp nao quebra a demo

Given que a Evolution API esteja indisponivel ou mal configurada,
When o pedido simulado for confirmado,
Then o usuario ainda deve ver a confirmacao do pedido e a falha deve ser apenas registrada no servidor.

### AC9 - Responsividade mobile-first

Given acesso em larguras de 375px, 390px e 414px,
When o usuario navegar pela demo,
Then nao deve haver overflow horizontal, textos sobrepostos ou controles inutilizaveis.

### AC10 - Qualidade minima antes de pitch

Given a demo implementada,
When os quality gates forem executados,
Then `npm run lint`, `npm run typecheck`, `npm test` e `npm run build` devem passar, e a demo deve estar pronta para validacao visual em mobile.

### AC11 - Referencia visual do exemplo HTML

Given que `exemplo.html` existe na raiz do projeto como referencia do cardapio desejado,
When a demo V1 for implementada em Next.js,
Then a experiencia, hierarquia visual, tema Vino e Cucina, estrutura de header, hero, categorias, cards, modal, toast e carrinho devem seguir o exemplo, adaptados para componentes React/Tailwind sem copiar dependencias ou scripts inline desnecessarios.

## Scope

### In Scope

- Bootstrap do app Next.js 14 com Tailwind CSS e TypeScript.
- Rota `/demo` para a demo estatica.
- Rota raiz redirecionando ou apontando para `/demo`.
- Dados hardcoded do restaurante, categorias e pratos.
- Componentes publicos do cardapio definidos no SDD.
- Estado local de carrinho com Zustand ou equivalente centralizado conforme SDD.
- Checkout simulado sem Mercado Pago real.
- API `/api/pedido-simulado` para validacao basica e tentativa de envio ao grupo WhatsApp All In.
- Formatadores e validadores necessarios para moeda BRL e telefone BR.
- Testes e validacao responsiva basica.
- `.env.example` atualizado com variaveis V1.
- Uso de `exemplo.html` como referencia visual e funcional para a demo Vino e Cucina.

### Out of Scope

- Supabase, banco de dados, RLS ou storage.
- Auth e dashboard admin.
- Multi-tenant por subdominio.
- Mercado Pago real, webhook real ou pagamento aprovado de verdade.
- Delivery completo.
- Persistencia de pedidos em banco.
- Integracao com WhatsApp do restaurante final.
- Landing page institucional da All In.
